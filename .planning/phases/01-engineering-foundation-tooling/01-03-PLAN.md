Create `.gitleaks.toml` at the repo root with EXACTLY this content (D-18, derived from OQ-6):

```toml
# .gitleaks.toml
title = "Base Layers Diagnostic — gitleaks config"

[extend]
# Extend the default gitleaks ruleset (covers AWS keys, GitHub tokens, generic API patterns, etc.)
useDefault = true

[[rules]]
id = "sha256-hex-literal-regression"
description = "SHA-256 hex literal (64 chars) in source — regression check for C2 INTERNAL_PASSWORD_HASH pattern"
regex = '''(?i)(password|hash|secret|key|token|credential)[^=\n]{0,20}[=:]\s*["\']?[a-f0-9]{64}["\']?'''
secretGroup = 0
tags = ["custom", "c2-regression"]
severity = "CRITICAL"

[allowlist]
description = "Allow legitimate hex strings in test fixtures and runbooks"
paths = [
  "tests/",
  "runbooks/",
]
regexes = [
  # Vitest snapshot hex is fine (e.g., expect(hash).toBe("...") in regression tests)
  '''expect\([^)]+\)\.toBe\(["\'][a-f0-9]{64}["\']''',
]
```

Load-bearing constraints:
- `(?i)` — case-insensitive match (catches `PASSWORD`, `Password`, `password`)
- Context-word prefix (`password|hash|secret|key|token|credential`) reduces false positives — random 64-char hex without a labeled context (e.g., a transient Firebase App Check token in a debug log) does NOT trigger
- `[^=\n]{0,20}` — up to 20 characters between context word and assignment, allowing `INTERNAL_PASSWORD_HASH = ...`
- `[a-f0-9]{64}` — exactly 64 hex chars (SHA-256 output length)
- Allowlist paths `tests/` and `runbooks/` allow legitimate fixtures (e.g., a regression test that asserts the hash shape)

**Important:** This rule will NOT remove the existing `INTERNAL_PASSWORD_HASH` from `app.js` (that lives on until AUTH-14 in Phase 6). The rule only blocks NEW commits from introducing secrets. The existing line is grandfathered in via the staged-only scan mode (`gitleaks protect --staged`).

**One-time false-positive check:** Run `npx gitleaks detect --source . --config .gitleaks.toml --verbose --report-path /tmp/gitleaks-baseline.json --no-git || true`. Inspect the report. Expected:

- The existing `INTERNAL_PASSWORD_HASH` line in `app.js` SHOULD be flagged. This is correct — it documents the gap that AUTH-14 closes. Add a note in the SUMMARY: "Pre-existing C2 finding flagged by baseline scan — closure scheduled in Phase 6 AUTH-14."
- Firebase API keys in `firebase-init.js` are NOT 64-char hex (they're shorter alphanumeric base64-like strings) — should NOT trigger the custom rule. They MAY trigger gitleaks default rules for "Generic API Key" — confirm and add allowlist entries if needed.
- Asset hashes / git SHAs in `package-lock.json` are 40-char SHAs (git) or longer base64 (npm integrity hashes) — not 64-char hex strings preceded by `password=` — should NOT trigger.

If false positives appear in `firebase-init.js` (Firebase web config IS public per Firebase docs — NOT a secret), add an allowlist entry:

**Step 2 — Create `.husky/pre-commit`:**

Create the file `.husky/pre-commit` (NOT inside `.husky/_/`) at the repo root with EXACTLY this content:

```sh
#!/usr/bin/env sh
```

Line-by-line rationale:
- Line 1: POSIX shebang. Works on Git for Windows / WSL / macOS / Linux. NEVER use a Windows-only path or `pwsh` shebang — would break for Mac/Linux contributors.
- Line 2: `npx lint-staged` — uses the `lint-staged` config from `package.json` (`"*.js": ["eslint --fix", "prettier --write"]`). Note `eslint --fix` does NOT carry `--max-warnings=0` (Pitfall E — pre-commit is dev fast-feedback; CI is authoritative).
- Line 3: `npx gitleaks protect --staged --config .gitleaks.toml` — `protect` is the staged-diff scan mode (faster than `detect` which scans full history). `--staged` confines the scan to files in the index. Non-zero exit aborts the commit.

On Unix-like systems, set executable permission: `chmod +x .husky/pre-commit`. On Windows, git tracks the executable bit via `core.fileMode` settings; the file is invokable via the husky runner regardless. If running on Windows, run `git update-index --chmod=+x .husky/pre-commit` to ensure the bit is set in the index.
