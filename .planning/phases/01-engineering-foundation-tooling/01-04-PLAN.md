**Step 0 - Verify gh CLI is authenticated:**

Run `gh auth status`. If unauthenticated, surface as a blocker - SHA fetch requires GitHub API access (anonymous rate limit is too low and may fail). The user must run `gh auth login` before continuing.

**Step 1 - Fetch the latest stable release for each Action and its commit SHA:**

For each Action below, identify the LATEST release tag (avoiding pre-releases / `next` / `beta`), then resolve that tag to its 40-char commit SHA. The `gh api` resolver is deterministic - same tag always resolves to the same SHA. The commands below use `--jq` to extract the SHA cleanly.

Five Actions to resolve (per RESEARCH.md ~lines 639-644):

```bash
# 1. actions/checkout
LATEST_TAG=$(gh release view --repo actions/checkout --json tagName --jq .tagName)
SHA=$(gh api "repos/actions/checkout/git/ref/tags/${LATEST_TAG}" --jq '.object.sha')
# If .object.type is "tag" (annotated tag), follow it:
TYPE=$(gh api "repos/actions/checkout/git/ref/tags/${LATEST_TAG}" --jq '.object.type')
if [ "$TYPE" = "tag" ]; then
  SHA=$(gh api "repos/actions/checkout/git/tags/${SHA}" --jq '.object.sha')
fi
echo "actions/checkout ${LATEST_TAG} ${SHA}" >> _tmp_action_shas.txt

# Repeat the EXACT SAME pattern for the other four:
# 2. actions/setup-node
# 3. actions/upload-artifact
# 4. google/osv-scanner-action
# 5. gitleaks/gitleaks-action
```

Use a small loop or repeat the block five times. Annotated-tag dereferencing (`if TYPE = tag`) is mandatory - many releases use annotated tags, and a naive `git/ref/tags/<tag>` returns the tag-object SHA, not the commit SHA. Pinning to a tag-object SHA in `uses:` does NOT work - GitHub Actions will fail with "ref not found".

After running, `_tmp_action_shas.txt` should contain five lines like:

```
actions/checkout v4.2.2 11bd71901bbe5b1630ceea73d27597364c9af683
actions/setup-node v4.1.0 39370e3970a6d050c480ffad4ff0ed4d3fdee5af
actions/upload-artifact v4.6.0 65c4c4a1ddee5b72f98dc3e02f09a3e8b4ef87dc
google/osv-scanner-action v1.8.5 63b7cb9cb68a3d7c42c88dd68a0fc7c47c8e1d98
gitleaks/gitleaks-action v2.3.6 9b35a74e8a32c63099bdc5d0f14fcc4b2ea3d07f
```

(The example values above match the RESEARCH.md skeleton placeholders - your actual values may differ; use what `gh api` returns at execution time.)

**Step 2 - Validate each SHA is 40 hex characters:**

```bash
while read -r REPO TAG SHA; do
  if ! echo "$SHA" | grep -qE '^[a-f0-9]{40}$'; then
    echo "INVALID SHA for $REPO@$TAG: $SHA"
    exit 1
  fi
done < _tmp_action_shas.txt
echo "All 5 SHAs are 40-char hex"
```

If any SHA fails this check, stop and surface - Pitfall G. Likely causes: anonymous rate limit hit (re-auth gh CLI), tag points to a branch ref (use a release tag, not a branch name), or `--jq` extracted the wrong field.
