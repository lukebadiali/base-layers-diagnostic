<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → GCP STS | OIDC token issued by GitHub identifies the workflow + repo + branch + commit; STS exchanges for short-lived GCP credentials. No long-lived JSON key in repo or secrets. |
| GitHub Actions runner → Firebase Hosting CDN edge (post-deploy) | `curl -I` over public Internet against a CDN-cached origin; only response headers are read; no auth required. |
| GitHub Actions runner → npm registry (functions/ install) | TLS-only via `npm ci` against locked `functions/package-lock.json`; supply-chain controls already mitigated by Phase 1 (T-1-01). |
| Workflow → main branch (push trigger) | The `if: github.ref == 'refs/heads/main'` predicate restricts deploy to main. PR runs hit the preview job only. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-1 | Tampering | Header drop / silent weakening — typo in firebase.json or Firebase Hosting silently dropping a header | mitigate | Post-deploy `curl -I` assertion in `deploy` job final step verifies all 9 expected headers are present in the LIVE response (not just the config) — matches the 9-header schema test in `tests/firebase-config.test.js` from 03-02-PLAN.md. Fails the run with `MISSING: <header>` if absent. The schema test catches config-time regressions; this step catches runtime/CDN-level regressions. |
| T-3-5 | Elevation of Privilege | OIDC service account over-privileged | accept (Phase 3 baseline; mitigate fully in Phase 7) | 03-PREFLIGHT.md `## OIDC SA Roles` records the actual grant. If Phase 1 D-23 used `roles/firebase.admin` (broad), the disposition is accepted for Phase 3 since the SA only deploys hosting + functions; Phase 7's per-function service accounts (FN-04) narrow this. |
| T-3-functions-mod-not-found | Tampering / Availability | Functions deploy uploads stale or absent compiled JS because `cd functions && npm ci && npm run build` was skipped | mitigate | Both `deploy` and `preview` jobs include the explicit `cd functions && npm ci && npm run build` step BEFORE `firebase deploy`. Pitfall 5 verified. |
| T-3-deploy-collision | Availability | Two simultaneous pushes to `main` race to deploy and one cancels the other mid-flight, leaving partial state | mitigate | Concurrency group `firebase-deploy-main` with `cancel-in-progress: false` queues sequential deploys (no cancellation of in-flight). Worst case: second deploy waits for first to finish. |
