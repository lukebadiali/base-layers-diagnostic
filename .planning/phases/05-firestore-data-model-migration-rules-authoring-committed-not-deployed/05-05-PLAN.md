# Phase 5 Subcollection Migration - Operator Runbook

**Authored:** 2026-05-08 (Phase 5 Wave 5)
**Owner:** Hugh (operator)
**Estimated execution time:** 30-60 min (depends on production data volume; expect <100 orgs)
**Risk profile:** HIGH - touches production Firestore data-at-rest. Bracketed by gcloud export (D-04) for rollback.

## 1. Prerequisites

- [ ] Phase 5 Waves 1-4 all merged to main; rules + migration script + data/* rewrites + H7/H8 fixes all green in CI
- [ ] firebase-admin@13.8.0 installed in root devDependencies (verify: `npm pkg get devDependencies.firebase-admin`)
- [ ] gcloud CLI installed; `gcloud --version` reports >= 470.0.0
- [ ] gcloud auth application-default login completed; `gcloud auth application-default print-access-token` returns a token
- [ ] Operator principal has roles/datastore.importExportAdmin on bedeveloped-base-layers project (verify: `gcloud projects get-iam-policy bedeveloped-base-layers --format='table(bindings.role,bindings.members)' | grep importExportAdmin`)
- [ ] Backup bucket exists: `gsutil ls gs://bedeveloped-base-layers-backups/` returns the bucket; if not, run `gsutil mb -l europe-west2 gs://bedeveloped-base-layers-backups`
- [ ] Storage Lifecycle policy on the bucket per Phase 8 BACKUP-02 (deferred - Phase 5 inherits whatever Phase 8 documents; out of scope for this runbook)
- [ ] No live writes happening (PROJECT.md "between client engagements" baseline; verify by checking Firebase Console -> Firestore Data tab for active sessions)

## 2. Pre-cutover Smoke Checklist

- [ ] `npm run test:rules` exits 0 in CI on the latest main commit
- [ ] `npm run test:migrate` exits 0 in CI on the latest main commit
- [ ] `npm test` (full unit suite) exits 0 - especially the 5-minute clock-skew test (ROADMAP SC#4) and the H8 dispatcher test
- [ ] `git log --grep="firebase deploy --only firestore:rules" -- .` returns empty (RULES-06 verification - no rules deploy commands in any Phase 5 commit)
- [ ] Production Firestore Console -> orgs/{anyOrgId} document inspector confirms current shape: nested-map under .responses, .comments, .actions, .documents, .messages, .readStates (NOT subcollections yet)

## 3. Cutover Steps

### 3.1 Take pre-migration backup (D-04 / Pitfall 10 closure)

```bash
ISO=$(date -u +%Y-%m-%dT%H-%M-%SZ)
gcloud firestore export gs://bedeveloped-base-layers-backups/pre-phase5-migration/$ISO/ \
  --project=bedeveloped-base-layers
```

Capture the operation name from the output (e.g. `projects/bedeveloped-base-layers/databases/(default)/operations/AS...`). The operation may take 1-10 minutes depending on data volume.

Wait for the operation to complete:
```bash
gcloud firestore operations describe <OPERATION_NAME> --project=bedeveloped-base-layers
```

`done: true` confirms the export landed. Verify the bucket has metadata:
```bash
gsutil ls gs://bedeveloped-base-layers-backups/pre-phase5-migration/$ISO/
```

Should list `firestore_export.overall_export_metadata` + per-collection metadata files. Record the ISO timestamp + bucket URI in 05-PREFLIGHT.md "Cutover Log".

### 3.2 Dry-run rehearsal (D-06)

```bash
cd /path/to/base-layers-diagnostic
node scripts/migrate-subcollections/run.js --dry-run 2>&1 | tee scripts/migrate-subcollections/dry-run-output.log
```

The script runs every step in dry-run mode: reads source orgs/{orgId} parent docs, builds target subcollection paths, logs `[DRY-RUN] would write N docs from orgs/<orgId>` per source doc + samples up to 5 target paths per step. ZERO Firestore writes; ZERO marker writes.

Eyeball the output:
- [ ] Each step (responses-v1, comments-v1, actions-v1, documents-v1, messages-v1, readStates-init) produces non-zero "would write" counts (zero suggests the source-shape inspection failed - investigate before proceeding)
- [ ] The pre-counts logged at the start match expectations (e.g., orgs count == known production org count)
- [ ] No `[WARN] unexpected shape` log lines (RESEARCH Open Question 2 - the dry-run is the discovery substrate)

If anything looks wrong, INVESTIGATE before proceeding to 3.3. The dry-run is your last cheap discovery point.

Commit the dry-run log:
```bash
git add scripts/migrate-subcollections/dry-run-output.log
git commit -m "chore(05-05): capture pre-cutover dry-run log evidence"
```

### 3.3 Real migration execution

```bash
node scripts/migrate-subcollections/run.js 2>&1 | tee scripts/migrate-subcollections/real-run-output.log
```

The script:
- Captures pre-counts (orgs.count + per-subcollection collectionGroup counts; expected zero for all subcollections)
- Iterates 6 STEPS (responses-v1 -> comments-v1 -> actions-v1 -> documents-v1 -> messages-v1 -> readStates-init - this is the Pitfall 10 leaves-first ordering)
- Per step: iterates every org doc; per source doc: writes pending marker -> batches 499 subcollection writes -> commits -> writes done marker
- Captures post-counts; runs assertCollectionGroupCount(preCounts, postCounts) - must succeed
- Runs assertFieldPresence(db, sampleSize=20) - must succeed (every sampled response/comment/message has legacyAppUserId/legacyAuthorId per D-03)
- Logs `[OK] Migration complete; pre/post assertions passed.`

If any assertion FAILS, the script exits with code 1. The migration is partially done; the markers under `migrations/{stepId}/items/{docId}` indicate what completed. Choose:
(A) Investigate the failure, fix the bug, re-run (idempotent - DONE markers skip; PENDING markers re-process; new sources get processed fresh).
(B) Roll back per section 5.

If `[OK]` logs without errors, proceed to section 4.

### 3.4 Commit the real-run log + post-cutover artefacts

```bash
git add scripts/migrate-subcollections/real-run-output.log
git commit -m "chore(05-05): capture phase 5 production migration execution log"
```

## 4. Post-Migration Verification

- [ ] Open Firebase Console -> Firestore -> orgs/{anyOrgId} -> Subcollections; confirm `responses`, `comments`, `actions`, `documents`, `messages`, `readStates` subcollections exist with non-zero doc counts
- [ ] Spot-check 3 migrated docs: pick a response, a comment, a message; verify each has `legacyAppUserId` (or `legacyAuthorId`) field set to the source `userId`/`authorId` value
- [ ] Run `node scripts/migrate-subcollections/run.js --verify` against production - exit code 0 means assertion harness still passes
- [ ] Run a quick read smoke test against production via the data/* wrappers (likely staged via dev: `npm run dev` + open localhost:5178; see if the dashboard loads + lists orgs + comments)
- [ ] Document the pre/post counts + spot-check evidence in 05-PREFLIGHT.md "Cutover Log" section

## 5. Rollback Procedure

**Trigger:** post-migration assertion harness fails OR data integrity issues discovered in section 4.

```bash
gcloud firestore import gs://bedeveloped-base-layers-backups/pre-phase5-migration/$ISO/ \
  --project=bedeveloped-base-layers
```

`gcloud firestore import` REPLACES production with the export taken at section 3.1. Allow 5-15 minutes for the operation. Verify rollback success:
- [ ] Spot-check orgs/{anyOrgId} - the parent doc has nested .responses/.comments/.actions/etc. fields again
- [ ] Subcollections under orgs/{anyOrgId} are empty (or absent)
- [ ] migrations/{stepId} markers may still exist - manually delete via Console: Firestore -> migrations -> Delete Collection

After rollback, RE-RUN the test suite + investigate the root cause before retrying. Do NOT redeploy rules or invoke the migration again until the root cause is understood.
