## Decisions Made

- **Recursion via `doInvite(true)` inside the confirmation modal's "Yes, reset" onclick** rather than splitting into two functions — keeps the disabled-button + error-surface plumbing in one place. The second inviteClient call uses a fresh clientReqId (generated inside the wrapper), so it's structurally distinct from the first call (not a replay against the idempotency-marker).
- **Defensive throw on unreachable outcome kinds** in inviteClient.ts — the passphrase-* outcomes can't reach the bottom of the function (Steps 1+2 throw before decideInviteOutcome is called with passphraseSet/Valid:false), but TypeScript narrows the discriminated union differently. The defensive throw is dead code in normal flow but cheap insurance against future refactors that break the precondition. Cost: +1 unreachable HttpsError site.
- **Admin SDK mock extension in place** rather than as a Wave-2-only seam — `getUserByEmail` / `createUser` / `updateUser` become first-class methods on `getAuthMock()` so future callables (resetClientPassword, etc.) reuse the same in-memory store. The `_seedUser` inspector is the test bridge.
- **Modal-flow test uses source-content pinning** (reads src/main.js content + grep-scopes to the `openInviteClientModal` function block) rather than IIFE bridge instrumentation. Consistent with the existing tests/main.test.js + tests/main/set-org-passphrase-policy.test.js pattern. The IIFE doesn't expose `openInviteClientModal` and adding a bridge would be more invasive than the test's benefit; a future wave that promotes the modal to an exported view module would unlock direct DOM-level assertions.
- **Integration test seeds state BEFORE loadWrapped()** and doesn't call `vi.resetModules()` inside loadWrapped — the admin-sdk.ts mock holds in-memory state at module scope, so a reset would discard the seed. setClaims integration test gets away with vi.resetModules because it doesn't pre-seed; this test must.
- **vi.hoisted spy in tests/cloud/invite-admin.test.js** — the original `const callableSpy = vi.fn()` declaration was referenced from inside the vi.mock factory at hoisted call time, causing a temporal-dead-zone error at side-effect import time of `src/observability/audit-events.js` (which the wrapper indirectly imports via src/firebase/auth.js → emitAuditEvent). Hoisting the spy declaration via `vi.hoisted` fixes it.

## Issues Encountered

- **node_modules not present in worktree at startup:** Both `node_modules` (root) and `functions/node_modules` were absent. The root one already had an existing junction-target (via prior wave setup); the functions one needed fresh `npm install`. Resolved via `cmd //c "mklink /J node_modules ..."` for the root junction, then `npm install --no-audit --no-fund` in the parent `functions/` directory (which the worktree junctions point at). 710 packages added in ~45s. Vitest then ran cleanly.
- **vi.mock factory hoisting / temporal-dead-zone error in tests/cloud/invite-admin.test.js:** Initial RED run failed with "Cannot access 'callableSpy' before initialization" because src/observability/audit-events.js (transitively imported via src/firebase/auth.js's emitAuditEvent re-export, which the wrapper closure pulls in) calls `httpsCallable(...)` at module-load time. vi.mock factories are hoisted above imports, so the factory tried to reference `callableSpy` (declared at module scope) BEFORE the const initializer ran. Fixed by hoisting the spy via `vi.hoisted(() => ({ callableSpy: vi.fn() }))` — the spy is then guaranteed initialized when the mock factory closes over it.
- **TypeScript narrowing through the discriminated union:** The decideInviteOutcome return type is a 5-variant union, but after Steps 1+2 throw, `passphrase-invalid` and `passphrase-not-set` are statically unreachable. TS can't prove this without a runtime guard, so `outcome.existingUid` at the resend branch failed type-check ("Property 'existingUid' does not exist on type ..."). Resolved by adding an explicit `if (outcome.kind !== "resend") throw ...;` narrow before the resend branch. The throw is dead code in normal flow.
- **Integration test seeded state was being lost by vi.resetModules:** First run of the integration test reported "Set the company passphrase first" on the happy-create path — the org doc seed wasn't visible to the loaded callable. Root cause: `loadWrapped()` called `vi.resetModules()` before importing the inviteClient, which re-imported `admin-sdk.js` fresh — but the test's beforeEach seeded the OLD instance's docStore Map. Fixed by removing the `vi.resetModules()` call (the admin-sdk module's in-memory state survives across t.wrap calls because vi.mock factories are stable; only the module's own state needs to be `_reset()`'d per test).
