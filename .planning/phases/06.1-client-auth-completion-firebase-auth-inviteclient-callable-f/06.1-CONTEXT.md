### inviteClient Callable Cloud Function

- **D-03:** **Input contract: `{email, name, orgId, orgPassphrase}`.** Zod-validated per Phase 7 FN-02 substrate. `orgPassphrase` is the plaintext the admin re-enters in the Invite modal; callable verifies it against `org.clientPassphraseHash` via `hashString(orgPassphrase) === org.clientPassphraseHash` BEFORE any user creation. If verification fails: throws `auth/passphrase-invalid` (mapped to friendly "Company passphrase incorrect — check it or update it via Set Passphrase" in the modal). If `org.clientPassphraseHash` is unset: throws `auth/passphrase-not-set` (mapped to "Set the company passphrase first" — same amber-banner condition the modal already handles for the legacy path).

- **D-04:** **Output contract: `{uid, existed: boolean}`.** No `tempPassword` field — admin already typed the passphrase that became the initial Auth password; surfacing it again is redundant. On `existed: false`: brand-new Auth user created; modal proceeds to `openInviteInstructionsModal` with revised copy. On `existed: true`: see D-06.

- **D-05:** **Bootstrap mechanism: org passphrase IS the initial Firebase Auth password.** `auth.createUser({email, password: orgPassphrase, emailVerified: true})` + `setCustomUserClaims(uid, {role: "client", orgId, firstRun: true})`. Every client invited to OrgAlpha is created with `password = <OrgAlpha's current passphrase>`. The passwordPolicy (≥12 chars + HIBP, per Phase 6 D-09) applies on createUser — operator who sets the org passphrase must use a passphrase that passes; existing `setOrgClientPassphrase` enforces this at passphrase-set time too (planner verifies/adds if missing). Plaintext exists ONLY transiently: in admin's browser memory while modal is open, in callable request body in transit (HTTPS), in callable memory during createUser, then gone. Never persisted server-side beyond the hash.

- **D-06:** **Idempotency: detect & confirm via the existed-flag.** Callable looks up by email via `auth.getUserByEmail(email).catch(() => null)`:
  - **Not found**: create the user as above. Return `{uid, existed: false}`.
  - **Found AND existing claims.orgId === requested orgId**: do NOT throw; return `{uid, existed: true, hasFirstRun: <bool>}`. Modal shows confirmation ("hugh+alpha@... already has a OrgAlpha account. Reset their password back to the current org passphrase and re-flip first-run?"). On admin confirm, modal calls `inviteClient` again with a `confirmReset: true` flag (or a separate `resendInvite` callable mode — planner picks). On confirm: `auth.updateUser(uid, {password: orgPassphrase})` + `setCustomUserClaims(uid, {role: "client", orgId, firstRun: true})`. Audit-event: `auth.client.invite.resend`.
  - **Found AND existing claims.orgId !== requested orgId**: refuse with `auth/cross-org-invite-rejected`. Mapped to "User already belongs to a different org — cross-org membership is not supported (AUTH-V2-02 deferred)." Audit-event still fires (`auth.client.invite.rejected.cross-org`) for OBS-05 anomaly counter.

- **D-07:** **Trust-on-invite email verification: `emailVerified: true` set server-side at createUser.** Admin vouches at invite time. AUTH-11 rules predicate (`request.auth.token.email_verified == true` for `orgs/{orgId}/*` access) passes without friction. Audit narrative: "client email is admin-vouched at invite time; binding is one-shot at invite, not periodically re-verified — credible-not-certified bar accepts this for low-privilege role with single-org blast radius." `renderEmailVerificationLanding` is NOT routed-to for clients (only internal admins who Console-create without operator-set `emailVerified: true` would hit it, and Phase 6 D-05 already sets the flag for them — so the view is essentially defensive substrate).

- **D-08:** **First-run mechanism: custom claim `{firstRun: true}` — REUSE existing internal-admin substrate.** `inviteClient` sets `firstRun: true` in the custom claim set. Router checks ID-token claim (`getIdTokenResult().claims.firstRun === true`) and routes to `renderFirstRun`. After password change, the existing `src/firebase/auth.js:347-362` updatePassword path calls `setClaims({uid, role, orgId})` which DROPS `firstRun` (because it's not in `SetClaimsSchema`) and forces ID-token refresh via `getIdToken(true)`. **Zero new client-side code for first-run handling.** Same mechanism, same code path, both roles. The Phase 6 follow-up BLOCKER-FIX 1 (reauth-on-password-change-before-setClaims to dodge `auth/user-token-expired`) inherits — clients hit the same loop-prevention guard.

### Client Sign-In + First-Run UX

## Specific Ideas

- **The org-passphrase-as-bootstrap-credential mechanism (D-05)** is the load-bearing UX decision that lets one passphrase invite N clients per org without per-user credential generation. Plaintext only exists transiently (admin's browser → callable in-flight → createUser call); never persisted server-side beyond the hash. The "admin re-enters passphrase at invite time" friction is essentially free because admin already knows the passphrase (they're about to share it OOB) and the re-entry doubles as a passphrase-verification gate ("if you forgot it, rotate it via Set Passphrase").
- **The first-run custom-claim mechanism (D-08)** is the dominant design choice here. Reusing the internal-admin substrate (setClaims drops `firstRun` + getIdToken refresh + Phase 6 BLOCKER-FIX 1 reauth guard) means zero new client-side code for first-run handling. One mechanism, both roles. Diverging here would have meant a second code path in `src/firebase/auth.js:347-362` updatePassword + a Firestore read for the router gate — pure substrate duplication.
- **Trust-on-invite emailVerified (D-07)** is honest substrate-honest disclosure: the binding is admin-vouched at one moment in time, not periodically re-verified. Audit narrative in DOC-10 § Client Authentication captures this verbatim. This deviates from the "click-to-verify" gold standard but the credible-not-certified bar accepts it for `role: client` (single-org blast radius + admin-vetted at invite + can be revoked by setClaims).
- **Optional opt-in MFA for clients (D-10)** matches the role-blast-radius threat model: internal admins can `setClaims` and read/write any org → hard MFA. Clients can only access their own org → optional. Cleanest audit narrative: "all users may enrol MFA; internal admin users MUST enrol; client users MAY enrol — per role-blast-radius threat model."
- **Cross-org invite refusal (D-06)** is the explicit AUTH-V2-02 boundary marker. Phase 06.1 does NOT support a single email being a client of two orgs. The error message points at the v2 deferral so future questionnaire-readers see the intentional scope decision.
- **Single atomic cutover commit (D-02)** is the cleanest audit-narrative for HANDOFF.md follow-up #9 closure: "the legacy state-machine.js + tests + setUserPassword + clientside sign-in branch all deleted in the same commit that wired up the new substrate; no dead-code window."
- **HANDOFF.md follow-up #3 (App Check 22h backoff)** is operationally relevant: if backoff is active during cutover, `inviteClient` returns `appCheck/throttled` and admin can't invite. Wave 1 pre-flight verifies App Check status; runbook documents the clear-site-data workaround OR the operator-paced fallback (delay invite until backoff clears). NOT in scope to fix the underlying reCAPTCHA Enterprise origin config — orthogonal Phase 7 substrate concern.
