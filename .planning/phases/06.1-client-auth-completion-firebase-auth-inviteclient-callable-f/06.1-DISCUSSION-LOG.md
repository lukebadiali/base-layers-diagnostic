## Invite mechanism & first sign-in

### Q1 — Primary invite mechanism

| Option | Description | Selected |
|---|---|---|
| Firebase email-link sign-in | inviteClient generates a one-time sign-in link via auth.generateSignInWithEmailLink; admin pastes link into mailto:; client lands signed-in and sets a password. No shared secret in transit. | ✗ (initially picked, then reversed) |
| Admin-set temp password (Phase 6 parity) | inviteClient generates a per-user random temp password; admin pastes into mailto:; client signs in, hits renderFirstRun. | ✗ (intermediate) |
| Password-reset link | inviteClient creates user with throwaway password + emailVerified:true, generates reset link via auth.generatePasswordResetLink; admin pastes link. | |

**User's choice:** Initially picked "Firebase email-link sign-in" but reversed mid-discussion: "I don't want an email auth system. I think that there should be a temp password set for client users that, when the user logs in, can reset their password for the first time."

**Final mechanism (clarified in Area 4):** Org passphrase is the shared initial credential — admin sets one passphrase per org, all clients of that org sign in with it first time. See Area 4 follow-up for the plaintext-handling resolution.

### Q3 — First-run flag mechanism

**First framing:**

| Option | Description | Selected |

**User's choice (initial):** "Firestore field users/{uid}.firstRun: true (Recommended)"

**Re-asked with corrected framing:** The "custom claim" option doesn't need a new callable — Phase 6 already wires this via setClaims (drops firstRun because not in SetClaimsSchema) + getIdToken(true) refresh in src/firebase/auth.js:347-362 updatePassword path. Internal admins ALREADY use this. Firestore field would create divergence between internal and client first-run handling.

| Option (re-asked) | Description | Selected |
|---|---|---|
| Custom claim {firstRun: true} — reuse existing internal-admin substrate | Zero new client-side code; one mechanism both roles. | ✓ |
| Firestore field users/{uid}.firstRun: true (original answer) | Diverges from internal-admin flow; second code path in updatePassword. | |

**User's choice (final):** "Custom claim {firstRun: true} — reuse existing internal-admin substrate"

**Notes:** Re-ask was necessary because the original Q3 framing misled the user about the substrate cost. Captured in CONTEXT.md D-08.
