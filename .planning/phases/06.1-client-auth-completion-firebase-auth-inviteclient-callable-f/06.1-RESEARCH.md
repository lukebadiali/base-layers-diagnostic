# Phase 06.1: Client auth completion — Firebase Auth + inviteClient callable - Research

**Researched:** 2026-05-22
**Domain:** Firebase Auth + Identity Platform server-side user provisioning; admin-initiated client onboarding; legacy substrate cutover
**Confidence:** HIGH (Phase 6/7 substrate already shipped; this phase reuses it verbatim plus one new callable)

## Summary

Phase 06.1 is a **mechanical extension of Phase 6's substrate** to the `role: "client"` user population. Internal admins already migrated; clients remain on legacy passphrase + per-user `passwordHash`. The work is small (~3 waves, one new callable + one new client wrapper + one modal edit + one atomic cutover commit) but carries several non-obvious load-bearing details that will trip the planner if missed.

The **dominant design choice** is reusing the existing `firebase/auth.js` `updatePassword`-with-BLOCKER-FIX-1 path for client first-run. The `firstRun: true` claim flips OFF when `setClaims({uid, role, orgId})` is called inside `updatePassword` — `SetClaimsSchema` does not include a `firstRun` field, so `setCustomUserClaims` overwrites the entire claim set and drops it. This means **zero new client-side code** for client first-run; same code path serves internal admins and clients.

**The single highest-impact verified finding from research:** Firebase Admin SDK `auth.createUser({password})` does **NOT** enforce the configured Identity Platform `passwordPolicy` (≥12 chars + HIBP). Policy is client-SDK-only. This means if an admin sets a weak org passphrase via `setOrgClientPassphrase`, `inviteClient` will succeed at `createUser` time — but the client will fail at `signInEmailPassword` with a policy-violation error. **The planner MUST add length + HIBP enforcement to `setOrgClientPassphrase` BEFORE Wave 2 lands**, OR document the failure mode in the runbook.

**Primary recommendation:** 3-wave shape per CONTEXT.md D-01; reuse `claims-admin-sa` (do not provision a new SA); add `auditEventInput` enum entries verbatim per D-11; verify `setOrgClientPassphrase` length-enforcement in Wave 1 pre-flight; defer integration test to a future TEST-09 follow-up; planner authors `confirmReset: true` as a parameter on `inviteClient` (not a separate callable) for simpler audit narrative.

## Critical Pinned Facts

### 1. Admin SDK `auth.createUser({email, password, emailVerified})` semantics

**1.1 `passwordPolicy` is NOT enforced at `createUser` time.** [VERIFIED: web search 2026-05-22 + GitHub firebase-admin-node issue #1141]

Identity Platform's `passwordPolicy` (Phase 6 D-09: ≥12 chars + HIBP) is **client-SDK-only enforcement**. It runs in the Firebase JS SDK at `signInWithEmailAndPassword` and `createUserWithEmailAndPassword` call sites. The Admin SDK bypasses all client-side policy because it uses a service account with full project access ("requests from the Firebase Admin SDK are not gated by security rules, as the Admin SDK is initialized with a service account which gives it full access").

**Operational consequence:** If `setOrgClientPassphrase` allows a weak passphrase (e.g., 6 chars), `auth.createUser({password: orgPassphrase})` succeeds. The client's first sign-in attempt then fails at the client SDK with `auth/password-does-not-meet-requirements` — and the user is bricked because there's no way for them to reset to a compliant password without admin intervention.

**Mitigation (planner adds in Wave 1 pre-flight or Wave 2):** Extend `setOrgClientPassphrase` to enforce length ≥ 12 + HIBP check BEFORE hashing. Two implementation paths:
- **Path A (recommended):** Client-side validation in the Set Passphrase modal — uses Firebase Auth's `validatePassword(auth, password)` API. Same enforcement surface as `signInWithEmailAndPassword`. Synchronous + cheap.
- **Path B (alternative):** Server-side validation via a new `setOrgClientPassphrase` callable. More substrate; covered by HANDOFF.md follow-up #1 (full-org-write cleanup) which is out of scope here.

**Recommendation:** Path A in Wave 1 — small modal-submit handler change in `src/main.js:5074+` (Set/Change passphrase modal location). Sized at ~15 lines.

**1.2 `emailVerified: true` IS honoured at `createUser` time.** [CITED: https://firebase.google.com/docs/auth/admin/manage-users]

The Admin SDK `createUser({email, password, emailVerified: true, displayName})` accepts and persists the flag verbatim. The user is created with `emailVerified === true` from the first ID token onwards. AUTH-11 rules predicate (`request.auth.token.email_verified == true`) passes without friction. `renderEmailVerificationLanding` is **not routed-to** for clients (D-07 trust-on-invite); the view remains as defensive substrate from Phase 6 D-14.

**1.3 Error code surface for `createUser`:** [CITED: https://firebase.google.com/docs/auth/admin/errors]
- `auth/email-already-exists` — handled by D-06 idempotency path (lookup-by-email runs FIRST so this should never fire; if it does, race condition mitigation is the idempotency-marker doc).
- `auth/invalid-email` — caught by Zod validation upstream; never reaches `createUser`.
- `auth/invalid-password` — Admin SDK returns this for **structurally invalid** passwords (empty string, non-string). Does NOT cover passwordPolicy compliance (per §1.1). Caught upstream by Zod (`.min(1)`).
- `auth/operation-not-allowed` — Email/Password sign-in disabled at project level. Not expected post-Phase-6.

**Error mapping for client `src/cloud/invite-admin.js`:**
- `permission-denied` → admin role required. Should never happen on a working admin path.
- `auth/passphrase-invalid` → throw `PassphraseInvalidError` ("Company passphrase incorrect — check it or update it via Set Passphrase").
- `auth/cross-org-invite-rejected` → throw `CrossOrgError` ("User already belongs to a different org. Cross-org membership is not supported.").
- `auth/passphrase-not-set` → throw `PassphraseNotSetError` ("Set the company passphrase first via the org Settings page.").
- Other → re-throw as generic `Error` so Sentry catches; modal surfaces "Something went wrong — try again or check the runbook."

### 3. `auth.getUserByEmail` race + idempotency

[VERIFIED: web search 2026-05-22 + GitHub firebase-admin-node]

**Error semantics:** `getUserByEmail(email)` throws (rejects the promise) with `error.code === "auth/user-not-found"` when no user matches. There is no `null` return value — it's always either a `UserRecord` or a thrown error.

**Recommended pattern (verbatim per CONTEXT.md D-06):**

```typescript
let existingUser: import("firebase-admin/auth").UserRecord | null = null;
try {
  existingUser = await getAuth().getUserByEmail(data.email);
} catch (err) {
  const code = (err as { code?: string })?.code ?? "";
  if (code !== "auth/user-not-found") {
    // Non-fatal codes: re-throw so Sentry catches + caller sees 500.
    // Fatal codes typically only appear under outage scenarios; rare.
    throw new HttpsError("internal", `Auth lookup failed: ${code}`);
  }
  // user-not-found is the happy path for new invites.
}
```

**Catch only `auth/user-not-found`. Re-throw everything else** so network/permission/SA-not-bound errors don't get silently treated as "user doesn't exist" → cascading account-creation behaviour.

**Concurrent-invite race (idempotency-marker keying):** The `ensureIdempotent` key MUST be keyed on the `clientReqId` from the client (matches Phase 7 FN-03 substrate convention) PLUS the email — the 5-minute window then absorbs concurrent same-email same-clientReqId replays as no-ops. If two admins invite the same email concurrently with DIFFERENT clientReqIds, the idempotency-marker doesn't dedup them — but the `auth.getUserByEmail` lookup on the second invite will find the first one's just-created user and route through the existed-flag branch (D-06). Functional correctness preserved.

### 4. App Check 22h backoff (HANDOFF.md follow-up #3)

[VERIFIED: GitHub firebase-js-sdk issue #7371 + #6708 + #9135]

**Mechanism:** When App Check fails to obtain a token (typically because reCAPTCHA Enterprise origin config is wrong, OR the site key doesn't match the deployed origin, OR the user's browser fails reCAPTCHA in a way that returns 403), the App Check SDK enters a **24-hour backoff** that returns `appCheck/throttled` on every subsequent token-fetch attempt. During backoff, all callables with `enforceAppCheck: true` return `unauthenticated` errors on the server side.

**Scope:** Per-browser. Clearing site data on `baselayers.bedeveloped.com` in DevTools resets the SDK backoff. But it won't fix the underlying 403 if the reCAPTCHA Enterprise origin is misconfigured — the next token-fetch will fail again and re-enter backoff.

**Operational impact for Phase 06.1:** If `inviteClient` is invoked while the admin's browser is in App Check backoff, the call fails with `appCheck/throttled` BEFORE reaching the server. The modal must surface a friendly message ("Security check failed — try clearing site data and signing in again, or contact your administrator.").

**Recommendation for Wave 1 pre-flight:** Verify admin's browser is NOT in backoff before the first invite. The CONTEXT.md `<specifics>` flags this. Runbook addition: if backoff active, paste the DevTools `Application → Clear storage → Clear site data` instruction.

**NOT in scope:** Fixing the underlying reCAPTCHA Enterprise origin config — that's HANDOFF.md follow-up #3 / Phase 7 substrate concern. Phase 06.1 inherits the existing App Check substrate.

### 8. Cross-org refusal precision (D-06)

**Inviting admin's `orgId` source:** The callable reads `context.auth.token.orgId` from the verified ID token. Admins typically have `orgId: null` (system-wide access), so the predicate is asymmetric:

```typescript
// Cross-org refusal precondition:
const existingClaims = existingUser?.customClaims ?? {};
const existingOrgId = (existingClaims.orgId as string | null) ?? null;
const existingRole = (existingClaims.role as string | null) ?? null;

if (existingUser) {
  // Existed-flag branch (D-06):
  if (existingRole === "client" && existingOrgId !== null && existingOrgId !== data.orgId) {
    // Cross-org refusal.
    await writeAuditEvent({ type: "auth.client.invite.rejected.cross-org", ... });
    throw new HttpsError(
      "failed-precondition",
      `User already belongs to a different org (existing orgId=${existingOrgId}, requested=${data.orgId})`,
    );
  }
  // ... rest of existed-flag branch (D-06 same-org reset path) ...
}
```

**Important nuance:** The refusal is based on **the existing user's `claims.orgId`**, NOT the inviting admin's `orgId`. Admins (`role: "admin"`) typically have `orgId: null` — they can invite into any org. The `requested orgId` comes from the Invite modal's org select dropdown. The refusal fires only if the existing user is already bound to a DIFFERENT client org.

**Edge case: user exists in Auth but has NO custom claims yet.** Possible if someone was Console-created without going through `beforeUserCreated` (e.g., manually added before Phase 6 cutover). In this case `existingClaims === {}` → `existingOrgId === null` → cross-org refusal does NOT fire. The existed-flag branch treats this user as "claimable" and assigns claims via `setCustomUserClaims` + `updateUser({password: orgPassphrase})`. This is **correct behaviour** — it adopts the orphan user into the requested org. Document in DOC-10.

```typescript
// functions/src/auth/invite-builder.ts
// Phase 06.1 (AUTH-16 / D-12): pure transform helpers for the inviteClient
// callable's verify + branch logic. The Vitest unit-test seam per D-12.
// MUST NOT import firebase-functions/* or firebase-admin/*.

export interface OrgPassphraseGate {
  storedHash: string | null;
  candidate: string;
}

export type InviteOutcome =
  | { kind: "create" }
  | { kind: "resend"; existingUid: string }
  | { kind: "cross-org-refuse"; existingUid: string; existingOrgId: string }
  | { kind: "passphrase-invalid" }
  | { kind: "passphrase-not-set" };

export async function verifyOrgPassphrase(
  storedHash: string | null,
  candidate: string,
  hashFn: (s: string) => Promise<string>,
): Promise<boolean> {
  if (!storedHash) return false;
  const candidateHash = await hashFn(candidate);
  return candidateHash === storedHash;
}

export function buildInviteClaims(orgId: string): {
  role: "client";
  orgId: string;
  firstRun: true;
} {
  return { role: "client", orgId, firstRun: true };
}

export function decideInviteOutcome(args: {
  passphraseValid: boolean;
  passphraseSet: boolean;
  existingUser: { uid: string; orgId: string | null; role: string | null } | null;
  requestedOrgId: string;
  confirmReset: boolean;
}): InviteOutcome {
  if (!args.passphraseSet) return { kind: "passphrase-not-set" };
  if (!args.passphraseValid) return { kind: "passphrase-invalid" };
  if (args.existingUser === null) return { kind: "create" };
  if (
    args.existingUser.role === "client" &&
    args.existingUser.orgId !== null &&
    args.existingUser.orgId !== args.requestedOrgId
  ) {
    return {
      kind: "cross-org-refuse",
      existingUid: args.existingUser.uid,
      existingOrgId: args.existingUser.orgId,
    };
  }
  return { kind: "resend", existingUid: args.existingUser.uid };
}
```

## Common Pitfalls

### Pitfall 1: passwordPolicy NOT enforced at createUser time (§1.1)

**What goes wrong:** Admin sets a weak org passphrase (e.g., "test123") via `setOrgClientPassphrase`. `inviteClient` succeeds — `auth.createUser({password: "test123"})` returns a UserRecord without error. Admin shares the passphrase OOB. Client attempts to sign in via `signInWithEmailAndPassword` — client SDK rejects with `auth/password-does-not-meet-requirements`. Client is bricked because their only credential is policy-non-compliant.

**Why it happens:** Admin SDK bypasses all client-side enforcement (verified per § Critical Pinned Fact 1.1). passwordPolicy is enforced ONLY at the client SDK boundary.

**How to avoid:** Add length ≥ 12 enforcement to `setOrgClientPassphrase` modal in Wave 1 (Path A — modal submit handler). Add HIBP check via Firebase Auth's `validatePassword(auth, password)` API if time permits; otherwise defer to a future polish wave. Document the mitigation in DOC-10 § Client Authentication.

**Warning signs:** Client first-sign-in fails with `auth/password-does-not-meet-requirements`. Sentry node captures from the AUTH-12 chokepoint show this code.

### Pitfall 2: `getUserByEmail` race on concurrent invites (§3)

**What goes wrong:** Two admins invite the same email concurrently with DIFFERENT clientReqIds. Admin A's `getUserByEmail` returns user-not-found; admin A's `createUser` succeeds. Admin B's `getUserByEmail` runs slightly later, returns admin A's just-created user. Admin B's modal shows the existed-flag confirmation ("reset their password back to the org passphrase?") which is correct but disconcerting if admins didn't coordinate.

**Why it happens:** `getUserByEmail` is not transactional with `createUser`. The 5-minute idempotency-marker doesn't help because clientReqIds differ.

**How to avoid:** Accept this as the expected behaviour. Document in runbook: "if two admins invite simultaneously, the second admin sees a confirmation modal asking to reset — this is correct; coordinate OOB to decide who proceeds." The existed-flag branch handles this gracefully — never creates duplicate users.

**Warning signs:** Audit log shows two `auth.client.invite` events for the same email within seconds; second is `auth.client.invite.resend`.

### Pitfall 3: BLOCKER-FIX 1 reauth on stale email (§6.3)

**What goes wrong:** Client signs in with org passphrase, lands on `renderFirstRun`. Client types new password; `updatePassword` succeeds; reauth path at `src/firebase/auth.js:332-345` calls `EmailAuthProvider.credential(user.email, newPassword)`. If `user.email` is null/empty (defensive guard), `reauthed` stays false and the setClaims+firstRun-flip block is skipped. Client gets bounced via `auth/user-token-expired` on the next ID-token-requiring call.

**Why it happens:** Existing edge-case in the BLOCKER-FIX 1 substrate. Phase 6 acknowledged but didn't fully close.

**How to avoid:** Verify in Wave 2 that `auth.createUser({email})` always populates the `email` field on the UserRecord (it does, per Admin SDK docs). The `user.email` guard at line 337 should never short-circuit for invited clients. Document the invariant.

**Warning signs:** Client lands on `renderFirstRun`, types new password, gets booted to sign-in. Sentry node captures `auth/user-token-expired` in the post-update flow.

### Pitfall 4: Cross-org refusal silently accepts orphan users (§8 edge case)

**What goes wrong:** User created via Firebase Console manually, never went through `beforeUserCreated`, has `customClaims === {}`. `inviteClient` does NOT refuse — assigns claims via the existed-flag branch.

**Why it happens:** The cross-org refusal predicate is `existingOrgId !== null && existingOrgId !== data.orgId`. An orphan user with `existingOrgId === null` passes the check.

**How to avoid:** Document this as intentional behaviour in DOC-10 — the orphan-adoption pathway is correct. **However:** add an audit event `auth.client.invite.adopt-orphan` (5th enum entry, planner's discretion) if forensics value exceeds the substrate cost. Recommendation: skip the 5th entry; emit `auth.client.invite.resend` for orphan adoption with a payload field `orphan: true`.

**Warning signs:** Audit log shows `auth.client.invite.resend` with `payload.orphan: true` for a user that should never have been Console-created.

### Pitfall 5: App Check 22h backoff during cutover (§4)

**What goes wrong:** Admin attempts first invite immediately after Wave 3 cutover lands. Browser is in App Check backoff from a prior Phase 7 substrate test. `inviteClient` callable rejects with `appCheck/throttled`.

**Why it happens:** HANDOFF.md follow-up #3; per-browser backoff lasts up to 24h.

**How to avoid:** Wave 1 pre-flight runbook step: "Clear site data on `baselayers.bedeveloped.com` before first invite if any callable call has failed in the last 24 hours." Add a one-line check to the cutover runbook.

**Warning signs:** Modal error surfaces "Security check failed" or callable returns `unauthenticated`. DevTools console shows `appCheck/throttled`.

### Pitfall 6: Pattern A idempotency-marker doc TTL drift (Pitfall 6 from PITFALLS.md)

**What goes wrong:** Admin clicks Invite, modal shows error (e.g., wrong passphrase), admin corrects + clicks again within 5 min with same clientReqId. Idempotency-marker doc still exists with old failure → `ensureIdempotent` throws `already-exists`.

**Why it happens:** `ensureIdempotent` writes the marker BEFORE the side effect, so failed attempts ALSO leave markers.

**How to avoid:** Client wrapper (`src/cloud/invite-admin.js`) generates a fresh `clientReqId` per call via `crypto.randomUUID()`. The marker for the failed attempt expires harmlessly after 5 min. The retried call gets a new key → no collision. **Same as Phase 7 `claims-admin.js` pattern.**

**Warning signs:** Modal shows "Duplicate request" error on retry. Resolution: refresh modal (regenerates clientReqId).
