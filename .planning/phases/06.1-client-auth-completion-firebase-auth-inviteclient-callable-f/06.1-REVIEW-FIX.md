### CR-01: `inviteClient` resend branch can demote and password-reset an existing internal/admin user with the same email

**Files modified:**

- `functions/src/auth/invite-builder.ts`
- `functions/src/auth/inviteClient.ts`
- `functions/test/auth/invite-builder.unit.test.ts`
- `functions/test/auth/inviteClient.unit.test.ts`
- `functions/test/auth/inviteClient.integration.test.ts`

**Commit:** `99a3295`

**Applied fix:**

1. Added a new `existing-privileged-user` variant to `InviteOutcome` in `invite-builder.ts`.
2. Inserted a check in `decideInviteOutcome` BEFORE the cross-org branch: any `existingUser.role ∈ {"admin", "internal"}` returns the new outcome regardless of orgId. The branch ordering matters — privileged refusal preempts both cross-org and resend.
3. Wired the new outcome in `inviteClient.ts` to:
   - `HttpsError("failed-precondition", "Email belongs to an existing privileged user", { code: "auth/email-belongs-to-privileged-user" })`
   - Audit emit on `auth.client.invite.rejected.cross-org` with payload `{ reason: "privileged-user", existingRole, requestedOrgId, email }`.
4. Added a `reason: "cross-org"` discriminator to the original cross-org audit payload so the two refusal paths are distinguishable in audit queries.
5. Added 3 unit-test cells in `invite-builder.unit.test.ts` (admin any-org, admin same-org, internal) and 2 cells in `inviteClient.unit.test.ts` (admin + confirmReset; internal same-org). Added one integration-test cell `Test 3b` in `inviteClient.integration.test.ts` that asserts NO `updateUser` / `setCustomUserClaims` mutation occurs on the admin victim.

**Audit-schema decision (per user instruction):** Reused the existing `auth.client.invite.rejected.cross-org` enum entry with a `payload.reason` discriminator rather than adding a new enum value. `auditEventSchema.ts` uses a constrained `z.enum()`; adding a new entry would churn downstream payload validators in audit-log readers (Phase 9+ anomaly rules). The discriminator keeps queries distinguishable without schema migration. Commit message documents the reasoning.

### WR-01: Client/server SHA-256 parity breaks on the client fallback path

**Files modified:**

- `src/util/hash.js`
- `tests/util/hash.test.js`

**Commit:** `de12583` (combined with WR-04 since both edit `src/util/hash.js` headers)

**Applied fix:**

Dropped the `try/catch` block entirely from `hashString`. The function now lets `crypto.subtle.digest` errors propagate to callers. Replaced the "falls back to a 32-bit hash string" test case with two new cases pinning sync-throw and async-rejection propagation (`rejects.toThrow`).

The `src/main.js#setOrgClientPassphrase` consumer invokes `hashString` from an async click handler without a try/catch — a thrown digest error now surfaces as an unhandled rejection (loud failure) rather than writing a divergent FNV string to `orgs/{orgId}.clientPassphraseHash`. This matches the WR-01 guidance ("let it propagate so callers see a loud failure rather than a silent parity break").

Coverage gate `src/util/** = 100%` was verified after the change (the removed catch-block branch left no untested code).
