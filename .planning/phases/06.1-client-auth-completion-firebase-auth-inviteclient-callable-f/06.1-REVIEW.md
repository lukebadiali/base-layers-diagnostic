## Summary

Phase 06.1 lands the `inviteClient` Cloud Function plus its client-side wrapper, the AUTH-12 chokepoint extensions, the cross-language SHA-256 parity helpers, the legacy substrate deletion, and the defensive `passwordHash` strip script. The architecture is sound, the TDD cadence is well documented, and the legacy-deletion sweep is clean — grep confirms zero references to `state-machine.js`, `setUserPassword`, `openChangePasswordModal`, `state.authTab`, or `u.passwordHash`/`user.passwordHash` in `src/` or `tests/` (the remaining mentions are comments documenting the removal).

The review surfaces one **Critical** finding (a non-cross-org role-takeover path in the resend branch), four **Warning** findings (a real client/server hash parity divergence on the SDK-failure fallback path; an unconditional `setCustomUserClaims` overwrite in the same resend path that may demote non-client users; a stale audit-emit narrative comment; and a stale `tests/util/hash.test.js` doc-comment about Phase 6 deleting the helper), plus four **Info** findings.

## Critical Issues

### CR-01: `inviteClient` resend branch can demote and password-reset an existing internal/admin user with the same email

**File:** `functions/src/auth/invite-builder.ts:88-114`, `functions/src/auth/inviteClient.ts:280-316`

**Issue:** `decideInviteOutcome` only returns `cross-org-refuse` when `existingUser.role === "client" && existingUser.orgId !== null && existingUser.orgId !== requestedOrgId`. If an admin (or internal) caller passes an email that already belongs to an existing `role: "internal"` or `role: "admin"` user — regardless of which org — the function falls through to the `resend` branch. With `confirmReset: true` the callable then runs:

```ts
await getAuth().updateUser(outcome.existingUid, { password: data.orgPassphrase });
await getAuth().setCustomUserClaims(outcome.existingUid, buildInviteClaims(data.orgId));
// → claims become { role: "client", orgId: data.orgId, firstRun: true }
```

This (a) overwrites the internal/admin user's Firebase Auth password to the shared org passphrase, and (b) demotes them to `role: "client"` in the requested org. The role-gate at lines 76-84 permits both `admin` and `internal` callers to invoke the function, so an `internal`-role staff member can target an `admin` colleague's email and silently turn their account into a client account — losing admin privileges and giving any client in that org's passphrase channel sign-in capability for that email.

The integration test (Test 2) only exercises `existing user with role:"client" + same org`. The unit-test "orphan claims (orgId null, role null)" case at `invite-builder.unit.test.ts:127-135` documents this branch as intentional ("RESEARCH § 8 adoption"), but the adoption logic does not distinguish "orphan with no role" from "existing privileged user".

**Fix:** Tighten `decideInviteOutcome` so any existing user whose role is `"admin"` or `"internal"` is refused (or returns a new `existing-privileged-user` outcome the callable converts to `HttpsError("failed-precondition")` with a distinct details.code). Only `role: null | "client"` should be eligible for the resend branch:

```ts
// invite-builder.ts
export type InviteOutcome =
  | { kind: "create" }
  | { kind: "resend"; existingUid: string }
  | { kind: "cross-org-refuse"; existingUid: string; existingOrgId: string }
  | { kind: "existing-privileged-user"; existingUid: string; existingRole: "admin" | "internal" }
  | { kind: "passphrase-invalid" }
  | { kind: "passphrase-not-set" };

// In decideInviteOutcome, before the cross-org branch:
if (args.existingUser.role === "admin" || args.existingUser.role === "internal") {
  return {
    kind: "existing-privileged-user",
    existingUid: args.existingUser.uid,
    existingRole: args.existingUser.role,
  };
}
```

Then `inviteClient.ts` maps `existing-privileged-user` to `HttpsError("failed-precondition", "Email belongs to an existing privileged user", { code: "auth/email-belongs-to-privileged-user" })` plus an audit emit on `auth.client.invite.rejected.cross-org` (or a new enum entry). Add a unit-test cell and an integration-test cell pinning the refusal.

## Warnings

### WR-01: Client/server SHA-256 parity breaks on the client fallback path

**File:** `src/util/hash.js:12-25`, `functions/src/util/hash.ts:19-21`, `functions/test/util/hash-parity.test.ts`

**Issue:** The client-side `hashString` wraps `crypto.subtle.digest` in a try/catch and, on failure, returns a base-10 string from a 32-bit FNV-style fold:

```js
} catch {
  let h = 0;
  for (const ch of String(s)) h = ((h << 5) - h + ch.charCodeAt(0)) | 0;
  return String(h);
}
```

The server-side `hashString` has no equivalent fallback — it returns SHA-256 hex unconditionally. The five `hash-parity.test.ts` vectors only pin happy-path output. If `crypto.subtle.digest` is ever unavailable on the admin's browser (legacy WebView, hostile policy, restricted iframe, locked-down enterprise device), `setOrgClientPassphrase` would store a 32-bit decimal string on `orgs/{orgId}.clientPassphraseHash`, and every subsequent `inviteClient` call would fail with `auth/passphrase-invalid` because the server SHA-256 hex of the same plaintext could never match. The drift is silent — no error surfaced at set-time, only at invite-time.

The `tests/util/hash.test.js` "falls back to a 32-bit hash string" cell (line 53-69) confirms the fallback is observable.

**Fix:** Either (a) drop the fallback now that the helper is load-bearing for inviteClient parity — let `setOrgClientPassphrase` throw when `crypto.subtle.digest` fails so the modal surfaces an error; OR (b) emit a console warning in the catch and route the modal to a "your browser does not support secure passphrase hashing" surface. Option (a) is the cleaner fix given Phase 06.1 makes this helper load-bearing:

```js
export async function hashString(s) {
  const enc = new TextEncoder().encode(String(s));
  const buf = await crypto.subtle.digest("SHA-256", enc);
  return Array.from(new Uint8Array(buf))
    .map((b) => b.toString(16).padStart(2, "0"))
    .join("");
}
```

Update `tests/util/hash.test.js` to remove the "falls back to a 32-bit hash string when crypto.subtle.digest throws" case (currently lines 53-69 — written for the Phase-2 demo-grade helper, not the Phase-06.1 load-bearing one).

### WR-02: Resend branch overwrites custom claims even when the existing user was already in the same org

**File:** `functions/src/auth/inviteClient.ts:298-310`

**Issue:** The `confirmReset` path unconditionally calls `setCustomUserClaims(existingUid, buildInviteClaims(data.orgId))` — i.e. it always sets `{ role: "client", orgId: data.orgId, firstRun: true }`. If the existing user already had additional claims (e.g. a future `mfaEnrolled` flag, a feature-flag claim, or an `email_verified`-related sub-claim — anything outside the three keys in `buildInviteClaims`), those are silently dropped on every confirmReset invite. This is the standard `setCustomUserClaims` overwrite-everything semantic (which is why `claim-builder.ts` exists for `setClaims`), so the issue is that the invite flow does not merge.

This is acceptable for Phase 06.1's three-claim shape, but the regression hazard is real: any future claim added in Phase 9+ will be wiped by `inviteClient.resend` unless the merge contract is added then.

**Fix:** Either accept and document the contract explicitly (e.g. comment block at `inviteClient.ts:298` saying "resend deliberately reverts the user to the canonical client claim shape — any future per-user claims must be re-applied after the resend completes"), or change to a read-merge-write:

```ts
const existingClaimsForMerge = (existingUser?.customClaims ?? {}) as Record<string, unknown>;
await getAuth().setCustomUserClaims(outcome.existingUid, {
  ...existingClaimsForMerge,
  ...buildInviteClaims(data.orgId), // overrides role/orgId/firstRun only
});
```

The first option is the cheaper close given the integration-test contract already pins the overwrite behaviour.

### WR-03: Stale audit-emit narrative — comment says `claim-builder.ts` drops `firstRun`; that file is not in scope here

**File:** `functions/src/auth/invite-builder.ts:48-57`

**Issue:** The `buildInviteClaims` JSDoc references `SetClaimsSchema` and `src/firebase/auth.js#updatePassword` to explain why `firstRun: true` must be the literal boolean true. This is correct in spirit, but the dependency chain (`SetClaimsSchema` lives in a different file from `buildInviteClaims`, `firstRun` is dropped at a different chokepoint, and the comment doesn't name the file) is hard to verify without grep. A future reviewer who reads `invite-builder.ts` in isolation cannot follow the reasoning to the closing point. Combined with WR-02, this elevates regression risk.

**Fix:** Add explicit file references in the JSDoc, e.g.:

```ts
/**
 * ...
 * RESEARCH § 6.2 regression-gate: `firstRun: true` MUST be the literal
 * boolean true. `functions/src/auth/claim-builder.ts` SetClaimsSchema
 * deliberately omits `firstRun`, so when `src/firebase/auth.js`
 * updatePassword() calls setClaims() post-password-change, the chained
 * `setCustomUserClaims` overwrites the entire claim set and drops
 * firstRun. That overwrite is what flips a first-run user OUT of
 * renderFirstRun on next getIdToken(true) refresh.
 * ...
 */
```

### WR-04: Stale doc-comment in `src/util/hash.js` says Phase 6 deletes it; it is now load-bearing for Phase 06.1

**File:** `src/util/hash.js:1-9`

**Issue:** The module header reads:

```js
// Phase 2 (D-05): byte-identical extraction from app.js:483-495.
// Phase 6 (AUTH-14) deletes this whole module — replaced by Firebase Auth.
```

That second sentence is now wrong — Phase 06.1 makes this module the client side of the cross-language hash-parity contract used by `setOrgClientPassphrase` and verified server-side by `inviteClient`. The same misleading "Phase 6 ... replaces with real Firebase Auth — this whole helper goes" line appears in the function JSDoc (lines 6-8). Anyone following the comment would delete the module and silently brick every future invite (server SHA-256 hex vs nothing on the client side).

**Fix:** Replace the misleading header with the load-bearing posture:

```js
// src/util/hash.js
// @ts-check
// Phase 2 (D-05): byte-identical extraction from app.js:483-495.
// Phase 06.1 (AUTH-16): LOAD-BEARING — client side of the cross-language
// SHA-256 hex parity contract with functions/src/util/hash.ts. Used by
// setOrgClientPassphrase to compute orgs/{orgId}.clientPassphraseHash;
// verified server-side by the inviteClient callable. See
// tests/util/hash.test.js + functions/test/util/hash-parity.test.ts.
```

Also drop the demo-grade "Trivial hashing for demo purposes (NOT secure — just to avoid plaintext storage)" function-level JSDoc — the helper is now part of the SHA-256 parity invariant and the comment incorrectly downgrades its security posture.
