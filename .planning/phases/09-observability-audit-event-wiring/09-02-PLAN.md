<objective>
Phase 9 Wave 2 — `@sentry/vite-plugin` source-map upload integration + CI env wiring + hidden-source-map gate.

Purpose: Source-map uploads to Sentry tag stack traces with the original source while NEVER serving the .map files publicly. The plugin runs only when `SENTRY_AUTH_TOKEN` is present (CI deploy/preview path); local dev and PR builds without the secret are no-ops. After upload, the .map files are deleted from `dist/` so Hosting cannot serve them. CI gates this invariant with an explicit `find dist -name "*.map"` count check (Pitfall 6 — hidden source maps).

Output: vite.config.js registers the plugin; CI build/deploy/preview jobs pass the env vars; an assertion step ensures the source-map cleanup runs (zero .map files served).

D-09-02: The plugin only fires when both `env.SENTRY_AUTH_TOKEN` is set AND `command === "build"`. PR/preview builds without the token are silent no-ops (mirrors the Phase 7 FN-07 reCAPTCHA placeholder pattern). This is the correct behaviour: PR previews don't need source maps in Sentry; only `main` deploys do.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI runner ↔ Sentry SaaS | CI uploads source maps via `@sentry/vite-plugin` using `SENTRY_AUTH_TOKEN`; auth token has `project:releases` + `project:write` scope (manage releases + upload sourcemaps); compromised token = malicious source map upload but no source data exfiltration |
| CI runner ↔ GitHub Secrets store | Secrets injected into the workflow only when running for `push: main` or `pull_request` (private repo, OIDC for cloud auth) |
| Firebase Hosting ↔ Public | Hosted bundle served to all browsers; if .map files leak, original source is publicly readable |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-9-02-1 | I (Information disclosure) | dist/**/*.map files served from Hosting | mitigate | `filesToDeleteAfterUpload: ["dist/**/*.map"]` in vite.config.js + post-build CI gate that fails the deploy if any .map remains. Two-layer defence (Pitfall 6). |
| T-9-02-2 | I (Information disclosure) | SENTRY_AUTH_TOKEN exposure in build logs | mitigate | GitHub Actions automatically masks secrets in stdout; `${{ secrets.SENTRY_AUTH_TOKEN }}` is the only source of truth (no echo, no `set -x`). The gate step does NOT print the token. |
| T-9-02-3 | T (Tampering) | Compromised SENTRY_AUTH_TOKEN | accept | Token scope is bounded to one Sentry project; if compromised, attacker can upload fake source maps but cannot read source data, modify Hosting, or invoke Cloud Functions. Operator-rotated quarterly per Phase 7 secret cadence. |
| T-9-02-4 | D (Denial of service) | Build failure when SENTRY_AUTH_TOKEN absent | accept | PR test builds run without the token (no-op plugin path). Deploy + preview gates require the token; missing token surfaces as a clear "FAIL: dist/*.map files present" message — operator action. No silent-fail. |
| T-9-02-5 | D (Denial of wallet) | Sentry plugin retries on transient API failure | accept | @sentry/vite-plugin v5 has built-in retry with backoff; CI build step has the standard `--max-old-space-size` headroom; plugin failure does not block hosting deploy unless we explicitly gate on it. The .map deletion gate fails the deploy independent of upload success — defence layer is local. |
</threat_model>
