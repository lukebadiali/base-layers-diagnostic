<objective>
Phase 9 Wave 5 — `authAnomalyAlert` Cloud Function (Slack dispatcher) + `authFailureCounters` rolling-window counter + firestore.rules block + synthetic-alert operator verification script + Slack-webhook gitleaks regex extension.

Purpose: A single `onDocumentCreated('auditLog/{eventId}')` trigger evaluates each new audit row against four anomaly rules (auth-failure burst >5/IP/5min, MFA disenrolment, role escalation to admin, unusual-hour `compliance.export.user`) and dispatches Slack messages via the `SLACK_WEBHOOK_URL` secret. Rule 1 uses a rolling-window counter at `authFailureCounters/{ipHash}` (same shape as Phase 7 `rateLimits/{uid}/buckets/{windowStart}`), with Slack firing EXACTLY ONCE when the threshold is first crossed (not on every event past threshold). The collection is server-only (rules deny all client read+write).

The 4 rules read from server-side bare audit emissions:
- Rule 1 (auth.signin.failure) — Plan 03a beforeUserSignedIn substrate; DORMANT until rejection rules exist
- Rule 2 (auth.mfa.unenrol) — DORMANT until MFA emit-site wiring lands (Plan 03a §mfa_rationale carry-forward)
- Rule 3 (iam.claims.set) — Plan 03a setClaims server emission; FUNCTIONAL
- Rule 4 (compliance.export.user) — Phase 8 baseline gdprExportUser server emission; FUNCTIONAL

The "DORMANT" designation means the trigger CODE is fully functional but the OBSERVATION pipeline emits zero rows today. When future code paths add the missing emit sources, the rules fire automatically with no schema or trigger-code change.

Output: New trigger module + index export + new Firestore collection + rules-emulator tests + operator verification script + .gitleaks.toml extension.
