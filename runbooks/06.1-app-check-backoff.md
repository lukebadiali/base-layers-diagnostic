# App Check 22h Backoff Workaround (Phase 06.1)

Operator runbook for the App Check SDK backoff scenario that prevents callable invocations (including `inviteClient`) while the admin browser is in the post-403 24-hour cooldown.

## Symptom

- `inviteClient` (or any other callable with `enforceAppCheck: true`) returns `unauthenticated` from the modal-submit handler.
- DevTools console shows `appCheck/throttled` (sometimes preceded by `appCheck/recaptcha-error`) on every token-fetch attempt.
- The Invite Client modal surfaces the AUTH-12 wrapper error mapped from the underlying App Check failure, e.g. "Security check failed — try clearing site data and signing in again, or contact your administrator."
- Time-since-symptom-first-observed is < 24 hours.

## Root cause

Per Phase 06.1 RESEARCH § 4: when App Check fails to obtain a token (typically because reCAPTCHA Enterprise origin config is wrong, OR the site key doesn't match the deployed origin, OR the user's browser fails reCAPTCHA in a way that returns 403), the App Check SDK enters a **24-hour backoff** that returns `appCheck/throttled` on every subsequent token-fetch attempt. During backoff, all callables with `enforceAppCheck: true` return `unauthenticated` errors on the server side.

The backoff is **per-browser**. Clearing site data on `baselayers.bedeveloped.com` in DevTools resets the SDK backoff. But it won't fix the underlying 403 if the reCAPTCHA Enterprise origin is misconfigured — the next token-fetch will fail again and re-enter backoff.

## Workaround (4-step DevTools Clear-site-data)

When the operator's admin browser hits the backoff in the middle of an invite flow:

1. Open DevTools (F12) on `baselayers.bedeveloped.com`.
2. Navigate to **Application** → **Storage** → **Clear site data**.
3. Tick **all** storage categories (Local + Session storage + IndexedDB + Cache + Service workers + Cookies).
4. Click **Clear site data**, then reload the page (Ctrl+R or Cmd+R).
5. Sign back in via Firebase Auth (email + password + MFA challenge if enrolled). Confirm `inviteClient` works.

If the underlying reCAPTCHA Enterprise origin config is the true root cause (HANDOFF.md follow-up #3), the next backoff will arrive within seconds-to-minutes of normal use. In that case escalate to F-06.1-01 forward-tracking row in `runbooks/phase-06.1-cleanup-ledger.md` — operator session on the Cloud Console reCAPTCHA Enterprise admin page.

## Verification

After the Clear-site-data step:

- DevTools Network tab: first request to `https://firebaseappcheck.googleapis.com/v1/projects/.../apps/.../exchangeRecaptchaEnterpriseDebugToken` (or `exchangeRecaptchaEnterpriseToken`) returns 200 within 5s of page load.
- DevTools Console: no `appCheck/throttled` warning after a 10s wait following sign-in.
- Test invite: open the Invite client modal, submit a happy-path invite (`hugh+uat-alpha@assume-ai.com`, org passphrase, etc.). Modal reports success (`existed: false, uid: ...`). Audit-log entry `auth.client.invite` lands in `auditLog/`.

## Out of scope

This runbook captures the **per-session operational mitigation**. The underlying reCAPTCHA Enterprise origin config fix is HANDOFF.md follow-up #3 / Phase 7 substrate concern. Phase 06.1 inherits the existing App Check substrate; fixing the underlying 403 root cause is queued as a forward-tracking row in `runbooks/phase-06.1-cleanup-ledger.md` row F-06.1-01.

If the App Check backoff bites repeatedly within a single operator session, escalate to:

1. Pause the in-flight invite flow.
2. Verify on the Firebase Console → App Check → Apps page that the `baselayers.bedeveloped.com` web app row shows the correct reCAPTCHA Enterprise provider + site key.
3. Verify on the GCP Console → Security → reCAPTCHA Enterprise → Keys page that the site key entry's "Allowed domains" list contains `baselayers.bedeveloped.com` exactly (not `bedeveloped.com` or `www.baselayers.bedeveloped.com`).
4. If misconfigured: file F-06.1-01 cleanup-ledger row update with the discovered config delta and pause invite work until origin alignment lands.
